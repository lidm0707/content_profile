# Plan: Supabase Database Initialization Tool

## Overview
Implemented CLI binary tool for initializing Supabase database tables from CSV schema definitions.

## Purpose
- Automate database table creation from schema files
- Support clean initialization with optional DROP TABLE
- Ensure consistent table structure across environments
- Enable rapid database setup for development and testing

## Implementation

### Files Created
- `src/bin/supabase_init.rs` - Main binary implementation
- `.supabase/tables/content.csv` - Content table schema
- `.supabase/tables/tags.csv` - Tags table schema  
- `.supabase/tables/content_tags.csv` - Junction table schema
- `.supabase/README.md` - Tool documentation

### Files Modified
- `Cargo.toml` - Added binary configuration and dependencies

### Dependencies Added
- `tokio` - Async runtime (1.40, features: full)
- `sqlx` - PostgreSQL client (0.8, features: postgres, runtime-tokio-rustls, chrono)
- `csv` - CSV parsing (1.3)
- `anyhow` - Error handling (1.0)

## Features Implemented

### Core Functionality
- Read CSV schema files from `.supabase/tables/` directory
- Parse table definitions (name, columns, data types, constraints)
- Generate SQL CREATE TABLE statements
- Execute SQL against PostgreSQL database
- Support for `--drop` flag to DROP TABLE CASCADE before recreation

### Schema Format
CSV files with columns: table_name, column_name, data_type, nullable, default_value, constraints

Supported constraint types:
- PRIMARY KEY GENERATED ALWAYS AS IDENTITY
- UNIQUE
- FOREIGN KEY REFERENCES
- CHECK constraints

## Usage

### Basic
```bash
export DATABASE_URL="postgresql://user:pass@host:port/db"
cargo run --bin supabase-init
```

### Clean Reinitialization
```bash
cargo run --bin supabase-init -- --drop
```

## Database Schema

### Tables
1. **content** - Blog posts/articles
   - id (PK, auto-increment)
   - title, slug (unique), body, status
   - created_at, updated_at, synced_at timestamps

2. **tags** - Hierarchical tags
   - id (PK, auto-increment)
   - name, slug (unique), parent_id (self-referencing FK)
   - created_at, updated_at, synced_at timestamps

3. **content_tags** - Many-to-many junction
   - id (PK, auto-increment)
   - content_id (FK -> content.id, CASCADE)
   - tag_id (FK -> tags.id, CASCADE)
   - UNIQUE(content_id, tag_id)
   - created_at timestamp
